# Dev Notes

- Review and refactor the `ACCEPTED` / `CACHE_READY` participant state split. The current flow should be reconsidered to decide whether these remain separate states or should be merged/simplified.
- Replace current shift polling with SSE so current shift state changes are pushed instead of periodically fetched.
- Review finding: the close flow now dead-ends when a shift reaches `READY_TO_COMMIT`. The backend still transitions to `READY_TO_COMMIT`, but `/api/shifts/current` filters that status out and the `/shifts/current` and `/shift-summary` routes sit behind a guard that only allows `READY_TO_START` / `IN_PROGRESS` / `CLOSE_REQUESTED` with `CACHE_READY` / `CLOSE_CONFIRMED`. Result: once the last participant confirms close, the lead can no longer finish the shift from the app.
- Review finding: the new flow strands users in `ACCEPTED` if cache prep or `/join-ready` fails. Shift creation inserts the lead as `ACCEPTED`, and the frontend only marks them join-ready after local snapshot rebuild succeeds. The home page only treats `INVITED` as a pending state, while the guarded current-shift routes reject `ACCEPTED`, so any interruption leaves an active shift visible but not recoverable from the UI.
- Review finding: declined participants still keep API access to shift details and live participant updates. `get_shift_waiting_room` and `subscribe_shift_events` only check for the existence of a `shift_participants` row, so retained `DECLINED` rows can still fetch the roster and subscribe to participant updates.
- `frontend/apps/main/src/pages/Index.tsx` appears unused and can likely be deleted. `App.tsx` routes `/` to `/home`, and there are no remaining imports or references to `Index.tsx` in the repository.
